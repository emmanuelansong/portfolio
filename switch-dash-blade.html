SwitchDashBlade

A browser-based top-down dungeon crawler built in Unity where the player fights through enemy-filled rooms using a weapon system that changes over time. The project explores how procedural weapon generation and dynamic difficulty adjustment can be combined to keep combat readable, varied, and challenging, while still delivering a polished 2D action game experience for desktop and web players.

// quick overview

What this project is about?

Summary

Switch-Dash-Blade is a 2D top-down dungeon crawler in which the player explores hostile rooms, avoids traps, defeats enemies, and progresses through a series of increasingly challenging levels. The core design twist is a procedurally generated weapon system that changes form and attributes during play, giving each combat phase a different feel and forcing the player to adapt. The project was designed to investigate how procedural content generation and dynamic difficulty adjustment could be expressed through a single combat mechanic. Instead of generating the entire game world, the team focused on weapon generation as the main system, using player performance metrics to influence the type, hilt, blade, speed, and damage of newly generated weapons.

My Role

I contributed as part of a six-person development team, helping deliver a playable Unity build through feature development, iteration, testing, and bug fixing. My work sat within a collaborative Agile workflow where the team split implementation into focused areas, reviewed progress in weekly scrums, and integrated code through shared version control. In addition to core development support, I contributed to gameplay polish and maintenance tasks, including fixes connected to environmental interactions and level usability. I also participated in project planning, reviews, testing, and release refinement, helping the team move from early prototypes to a public itch.io release.

Core Tools

Key Outcome

The final project delivered a fully playable browser and Windows build featuring procedurally generated weapons, adaptive difficulty, multi-room dungeon traversal, enemy encounters, traps, fog of war, menus, UI, and public playtest feedback. By release, the team had moved from a broad procedural generation concept to a clearer, more focused experience where weapon switching became the game’s defining system. The strongest outcome was proving that a relatively small team could build a polished combat-driven prototype while also applying structured software engineering practices, iterative testing, and public release feedback to improve balance and usability.

Systems & highlights

Gameplay Core Loop
Adaptive dungeon combat

The main gameplay loop centres on room-by-room dungeon combat. Players move through dark, trap-filled levels, clear enemies, pull levers to unlock progression, and survive long enough to benefit from their next generated weapon. Combat is intentionally readable and arcade-like, with light and heavy attacks, enemy pressure, room-based pacing, and environmental hazards such as spike traps and arrow traps. This structure gives the game a strong sense of progression while ensuring that every new room creates a short tactical problem for the player to solve.

Tech Architecture
Procedural weapons with dynamic difficulty

The project’s standout system is its procedural weapon generation pipeline. Every combat cycle, the game evaluates player performance using metrics such as damage dealt, enemies killed, time to clear rooms, and change in health. These values feed into a dynamic difficulty adjustment model that determines the next weapon’s type and overall power. Weapon parts are generated in a component-based way, allowing the game to mix attributes like damage, speed, and presentation while keeping output balanced. This made the procedural system more than a cosmetic gimmick: it actively shaped how the player approached each stretch of combat.

UX Polish
Readability, feedback & exploration

A major focus of polish was making the game’s systems legible to players. The UI communicates health and generated weapon stats, while fog of war hides unexplored rooms to improve tension and pacing. Sound, menus, movement tuning, and enemy adjustments were introduced late in development to improve the game feel and make the build more release-ready. Public and in-house playtesting helped identify issues around difficulty, visibility, collision, and progression. Those iterations made the final build feel more cohesive and much easier to understand moment to moment.

How it's built

Focus Areas

Procedural weapons, dynamic difficulty balancing, collaborative Unity development, release iteration, gameplay polish, and production workflow.

Phase 1
Planning & team structure
The team began by outlining the game’s core direction through a blend of Scrum-inspired Agile development and a playcentric design process. Early targets were organised through dependency diagrams and SMART objectives, allowing the project to map out what needed to exist before other systems could be built. Although the original plan leaned toward a Waterfall-style schedule, the team quickly recognised that a strict sequential process was inefficient for collaborative game development. Work shifted into a more parallel, iterative model where members could prototype, test, and integrate systems at the same time.
Phase 2
Prototype & core systems
Development first focused on creating a working playable game loop before expanding the procedural systems. The team built player movement, enemy encounters, room logic, UI, health, and basic levels inside Unity using C#. Once the core combat structure was stable, attention shifted to the procedural weapon generation system. Rather than procedurally generating every possible system, the project narrowed its scope to make weapons the centrepiece. This allowed the team to develop a more convincing and mechanically meaningful implementation of procedural generation.
Phase 3
Balancing, testing & bug fixing
Testing mixed manual regression testing, playtesting, and compatibility checks across different resolutions and builds. Because many game systems produced behaviour rather than simple return values, the team leaned heavily on in-engine validation, logging, and repeated hands-on testing rather than relying entirely on unit tests. This process surfaced issues such as enemy collisions, spike trap layering, UI layout concerns, and progression bugs. Public feedback also pushed the team to refine movement feel, add more useful attacks, improve audio, and smooth the experience between beta and full release.
Phase 4
Release & evaluation
The project was released first as a beta build on itch.io, followed by a fuller public release after bug fixing and gameplay improvements. This release pipeline let the team gather real user feedback, triage issues by priority and difficulty, and improve the build through targeted fixes. Looking back, the project showed the value of keeping scope focused, adapting process early, and using public feedback to shape the final product. The clearest next step would be expanding the procedural system further with more weapon modifiers, enemy variety, and additional build depth while preserving the readability of the current combat loop.
